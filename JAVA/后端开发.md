  方法二：直接在 sql 语句中起别名


## 主键返回

  1.在保存数据时有时候需要为一张表保存另一张表的主键，因此需要主键返回。

```java
  //在EmpMapper中先执行插入员工信息操作
  //然后需要再在EmpExprMapper中插入员工工作经历（需要于员工id绑定），因此需要主键返回
  
  //EmpMapper：
  @Options(useGeneratedKeys = true,keyProperty = "id")//获取自增ID
  @insert(...)
  public void insert(Emp emp);
  //service：
  public void save(Emp emp) {
      empMapper.insert(emp);
      List<EmpExpr> exprList = emp.getExprList();
      if(!CollectionUtils.isEmpty(exprList)){
          exprList.forEach(expr -> {// 设置员工ID
              expr.setEmpId(emp.getId());
          });
          empExprMapper.insertBatch(exprList);
      }
  }
  //EmpExprMapper：
  @insert(...)
  void insertBatch(List<EmpExpr> exprList);
```

## PageHelper分页

- 通过自动改造Sql语句来实现分页查询。

  ​	1.需要导入依赖：

```java
  <dependency>
  <groupId>com.github.pagehelper</groupId>
  <artifactId>pagehelper-spring-boot-starter</artifactId>
  <version>1.4.7</version>
  </dependency>
```

  ​	2.在Mapper层中定义sql语句只需要将数据查出来不需要再sql语句中分页：

```java
  //原版：
  @Select("select count(*) from emp e left join dept d on e.dept_id = d.id")
  public Long count();
  @Select("select e.*,d.name deptName from emp e left join dept d on e.dept_id = d.id " + "order by e.update_time desc limit #{start},#{pageSize}")
  public List<Emp> list(@Param("start") Integer start, @Param("pageSize") Integer pageSize);
  //pagehelper版：
  @Select("select e.*,d.name deptName from emp e left join dept d on e.dept_id = d.id order by e.update_time desc")
  public List< Emp> list();
```

  ​	3.再Service层中使用PageHelper：

```java
  @Override
  public PageResult<Emp> page(Integer page, Integer pageSize) {
      PageHelper.startPage(page,pageSize);//开始分页
      List<Emp> empList = empMapper.list();
      Page<Emp> p = (Page<Emp>)empList;//获取分页数据
      return new PageResult<Emp>(p.getTotal(), p.getResult());
  }
```

  ​	4.实现机制：先拦截sql，然后分两步，第一步记录总数，第二部分页查询然后把数据封装给Page<T>。

  ​		注意：- 在使用PageHelper中sql结尾不能加‘,’因为PageHelper会直接对sql改造所以不能加。

  ​			    - PageHelper仅仅能对紧跟在其后的第一个查询语句分页查询。

## Spring事务管理

  1.如果在一个业务方法中多次操作了数据库，可以在这个方法前加一个@Transactional注解，这样会自动实现事务的开始提交回滚。

  - rollbackFor属性用于控制出现何种异常类型，回滚事务。(默认情况下，只有出现RuntimeException才会回滚)
  - 如果在一个方法中，开启事务且用到try-final，出现异常final中的内容是会执行的，但如果final中执行了sql语句，这里面的sql语句也会被rollback，因此要使用事务的嵌套，为final中的方法另外开启一个事务且使用到propagation属性，包含以下常用属性：
    - REQUIRED:大部分场景
    - REQUIRES NEW:希望两个方法在独立的事务中运行，互不影响

  2.事务四大特性（ACID）：

  - 原子性：事务是不可分割的最小单元，要么全部成功，要么全部失败。
  - 一致性：事务完成时，必须使所有的数据都保持一致状态。
  - 隔离性：数据库系统提供的隔离机制，保证事务在不受外部并发操作影响的独立环境下运行 。
  - 持久性：事务一旦提交或回滚，它对数据库中的数据的改变就是永久的。

## 全局异常处理

  1. 全局异常处理需要两个关键的注释**@RestControllerAdvice**和**@ExceptionHandler**，其中@RestControllerAdvice=@ControllerAdvice+@ResponseBody（方法的返回值直接响应给前端）；@ExceptionHandler声明是个全局异常处理方法。
  2. 如果需要处理特定的错误则需要方法接收特定的错误类型，如下面代码中，DuplicateKeyException是Exception的一个子错误，在触发错误时优先执行子错误方法。

```java
  @Slf4j
  @RestControllerAdvice
  public class GlobalExceptionHandler {
      @ExceptionHandler
      public Result handleException(Exception e){//方法形参中指定能够处理的异常类型
          e.printStackTrace();//打印堆栈中的异常信息
          //捕获到异常之后，响应一个标准的Result
          return Result.error("对不起,操作失败,请联系管理员");
      }
      @ExceptionHandler
      public Result handleDuplicateKepException(DuplicateKeyException e){
          log.error("程序出错了",e);
          String message = e.getMessage();
          int index = message.indexOf("Duplicate entry");
          String errMsg = message.substring(index);
          String[] arr = errMsg.split(" ");
          return Result.error(arr[2] +"操作失败,该数据已存在");
      }
  
  }
```

## 登录、身份验证

### Session

- Session被内容被保存在服务器端，他是基于Cookie的前端的Cookie中会携带JSESSIONID，然后通过这个ID可以查到服务端保存的Session，在拦截器中有以下常规用法：

```java
public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
//根据当前HTTP请求中携带的JSESSIONID，从服务器的 session 存储中查找并返回对应的 HttpSession 对象。
    HttpSession session = request.getSession();
//从刚才拿到的、属于当前用户的session中，取出 key 为 "user" 的那个对象。
    Object user = session.getAttribute("user");
    if (user == null) {
        // 没有，需要拦截，设置状态码
        response.setStatus(401);
        // 拦截
        return false;
    }
    UserHolder.saveUser((User) user);
    // 有用户，则放行
    return true;
}
```

## AOP面向切面编程：

0. **why：**首先要弄清楚为什么要用AOP，因为有些功能比如统计方法用时，如果在每个方法中都进行统计，那就太麻烦了，所有用到AOP把这些方法所共用的功能整合到AOP方法中，使实现起来更加简便，逻辑性也强。

1. **AOP使用步骤：**分为两步，先要引入依赖，然后定义AOP方法并且@Aspect和@Component注释。其中@Around注释中定义了哪些方法会执行AOP方法。下面介绍一些概念：

- 连接点:JoinPoint，可以被AOP控制的方法(暗含方法执行时的相关信息）。
- 通知:Advice，指那些重复的逻辑，也就是共性功能(最终体现为一个方法)。
- 切入点:PointCut，在连接点中被AOP方法实际控制的方法。
- 切面:Aspect，描述通知与切入点的对应关系(通知+切入点)。
- 目标对象:Target，通知所应用的对象。

<img src="./后端开发.assets/image-20251106134735061.png" alt="image-20251106134735061" style="zoom:50%;" />

2. **底层逻辑：**AOP的底层是通过动态代理实现的，AOP会对目标对象生成代理对象，在代理对象中不仅执行目标对象的方法还要执行AOP类中所定义的方法，然后将代理对象放到容器中，controller在实际调用时调用的是代理对象。下面介绍一些不同注解：

- **@Around:**环绕通知，此注解标注的通知方法在目标方法前、后都被执行
  - @Around环绕通知需要自己调用 ProceedingJoinPoint.proceed()来让原始方法执行，其他通知不需要考虑目标方法执行
  - @Around环绕通知方法的返回值，必须指定为0bject，来接收原始方法的返回值。
- @Before:前置通知，此注解标注的通知方法在目标方法前被执行
- @After :后置通知，此注解标注的通知方法在目标方法后被执行，无论是否有异常都会执行
- @AfterReturning:返回后通知，此注解标注的通知方法在目标方法后被执行，有异常不会执行
- @AfterThrowing: 异常后通知，此注解标注的通知方法发生异常后执行

## ThreadLocal线程局部变量：

1. **what：**ThreadLocal其实是线程的局部变量，为每个线程提供单独一份存储空间，具有线程隔离的效果，不同的线程之间不会相互干扰。
2. **why（同线程数据共享）：**如果一个数据在Filter中拿到了（例如id），在AOP中要用则可以用ThreadLocal来保存，然后在AOP中使用。

![image-20251106164015477](./后端开发.assets/image-20251106164015477.png)

3. **使用方法：**定义一个CurrentHolder类来保存要用到的方法，然后在Filter中调用setCurrentId方法保存id，在要使用时调用getCurrentId()获取id，最后在Filter中调用remove()清理id。

```java
public class CurrentHolder {
    private static final ThreadLocal<Integer> CURRENT_LOCAL = new ThreadLocal<>();
    //线程绑定员工ID
    public static void setCurrentId(Integer employeeId) {
        CURRENT_LOCAL.set(employeeId);
    }
    //获取员工ID
    public static Integer getCurrentId() {
        return CURRENT_LOCAL.get();
    }
    //删除员工ID
    public static void remove() {
        CURRENT_LOCAL.remove();
    }
}
```



## Swagger——自动生成接口文档：

- 使用Swagger你只需要按照它的规范去定义接口及接口相关的信息，就可以做到生成接口文档，以及在线接口调试页面。
- 使用步骤：

1. 导入knife4j的maven坐标；
2. 在配置类中加入 knife4j 相关配置；
3. 设置静态资源映射，否则接口文档页面无法访问；

```java
//2-3步示例代码
/**
     * 通过knife4j生成接口文档
     * @return
     */
@Bean
public Docket docket() {
    ApiInfo apiInfo = new ApiInfoBuilder()
        .title("苍穹外卖项目接口文档")
        .version("2.0")
        .description("苍穹外卖项目接口文档")
        .build();
    Docket docket = new Docket(DocumentationType.SWAGGER_2)
        .apiInfo(apiInfo)
        .select()
        .apis(RequestHandlerSelectors.basePackage("com.sky.controller"))
        .paths(PathSelectors.any())
        .build();
    return docket;
}
/**
     * 设置静态资源映射
     * @param registry
     */
protected void addResourceHandlers(ResourceHandlerRegistry registry) {
    registry.addResourceHandler("/doc.html").addResourceLocations("classpath:/META-INF/resources/");
    registry.addResourceHandler("/webjars/**").addResourceLocations("classpath:/META-INF/resources/webjars/");
}
```

## 消息转换器：

- 数据库查询出的消息，不一定是需要的格式，因此可以配置一个全局的消息转换器来转换消息格式（例如时间）。需要在继承了WebMvcConfigurationSupport的配置类中编写一个格式化配置：

```java
@Override
protected void extendMessageConverters(List<HttpMessageConverter<?>> converters) {
    //创建一个消息转换器
    MappingJackson2HttpMessageConverter converter = new MappingJackson2HttpMessageConverter();
    //需要为消息转换器设置一个对象映射器，才能将java对象转为json
    converter.setObjectMapper(new JacksonObjectMapper());
    //将消息转换器对象添加到converters容器中
    converters.add(0,converter);
}
```

```java
 * WebSocket服务

@Component
@ServerEndpoint("/ws/{sid}")
public class WebSocketServer {
    //存放会话对象
    private static Map<String, Session> sessionMap = new HashMap();

    /**
     * 连接建立成功调用的方法
     */
    @OnOpen
    public void onOpen(Session session, @PathParam("sid") String sid) {
        System.out.println("客户端：" + sid + "建立连接");
        sessionMap.put(sid, session);
    }
    /**
     * 收到客户端消息后调用的方法
     *
     * @param message 客户端发送过来的消息
     */
    @OnMessage
    public void onMessage(String message, @PathParam("sid") String sid) {
        System.out.println("收到来自客户端：" + sid + "的信息:" + message);
    }

    /**
     * 连接关闭调用的方法
     *
     * @param sid
     */
    @OnClose
    public void onClose(@PathParam("sid") String sid) {
        System.out.println("连接断开:" + sid);
        sessionMap.remove(sid);
    }

    /**
     * 群发
     *
     * @param message
     */
    public void sendToAllClient(String message) {
        Collection<Session> sessions = sessionMap.values();
        for (Session session : sessions) {
            try {
                //服务器向客户端发送消息
                session.getBasicRemote().sendText(message);
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }

}
```

### Redis自增：

**Redis自增ID策略:** 

- 每天一个key，方便统计订单量，且正常情况下不会溢出。
- ID构造是时间戳+计数器。

**代码示例：**

```java
@Component
public class RedisIdWorker {
    // 开始时间戳
    private static final long BEGIN_TIMESTAMP = 1767225600L;
    // 序列号位数
    private static final int COUNT_BITS = 32;
    @Autowired
    private StringRedisTemplate stringRedisTemplate;
    public Long nextId(String keyPrefix) {
        LocalDateTime now = LocalDateTime.now();
        // 获取当前时间戳，单位秒
        long timestamp = now.toEpochSecond(ZoneOffset.UTC) - BEGIN_TIMESTAMP;
        // 从redis获取当前日期的序列号
        String date = now.format(DateTimeFormatter.ofPattern("yyyyMMdd"));
        Long sequence = stringRedisTemplate.opsForValue().increment("icr:" + keyPrefix + ":" + date);
        // 拼接id
        return timestamp << COUNT_BITS | sequence;
    }

}
```

## Feed流：

### Timeline:

- 不做内容筛选，简单的按照内容发布时间排序，常用于好友或关注。例如朋友圈

1. **拉模式（读扩撒）：**发送消息的人有个发件箱（可以带有时间戳），收消息的人每次从关注列表中分别读取这些发件箱。缺点是每次都要读取效率慢。——很少使用
2. **推模式（写扩散）：**每个粉丝有个收件箱，发消息的人发消息之后会直接发给每一个粉丝的收件箱。缺点是内存占用高。——适合用户量少
3. **推拉结合（读写混合模式）：**大博主会有发件箱，普通人只会有收件箱。大博主发送的消息会发在发件箱内（方便不活跃粉丝读取），并且发到活跃用户的收件箱。而普通用户没有发件箱直接发到所有粉丝的收件箱。——适合有大V用户量大

### 智能排序:

- 利用智能算法屏蔽掉违规的、用户不感兴趣的内容。推送用户感兴趣信息来吸引用户

### 滚动分页：

- 因为Feed流中的数据会不断更新，所以数据的角标也在变化，因此不能采用传统的分页模式。滚动分页就是记住了上一次查询的lastId来继续查询下一页。
- 使用Redis中的ZSet就可以实现这一点，因为这个集合中每个数据有有个分数（可以设置为时间戳），查询下一页时可以查询比上一页最后一条信息时间小的信息。

# 消息队列

## 基于Redis消息队列

1. **List实现：**利用list中的LPUSH结合 BRPOP、或者 RPUSH结合BLPOP来实现。

**优点:**利用Redis存储，不受限于JVM内存上限；基于Redis的持久化机制，数据安全性有保证；可以满足消息有序性。

**缺点:**无法避免消息丢失（无法避免在消息被取走之后丢失）；只支持单消费者。



2. **PubSub实现：**使用Redis中的订阅和发布功能实现消息队列（可以SUBSCRIBE多个PUBLISH）。

**优点:**采用发布订阅模型，支持多生产、多消费

**缺点:**不支持数据持久化（pub的消息不会保存在redis）；无法避免消息丢失消息；堆积有上限，超出时数据丢失。



3. **Stream流实现：**使用Redis5.0最新的Stream流进行写入和读取消息。

- **单消费者：**也就是单个监听，创建一个持久化队列然后在里面读取消息，但不知道消息是否被读过或者成功执行。

**优点：**消息可回溯；一个消息可以被多个消费者读取；可以阻塞读取。
	**缺点：**有消息漏读的风险。

- **消费者组（推荐做法）:**将多个消费者划分到一个组中，监听同一个队列。在黑马点评VoucherOrderServiceImpl中有实现
- **特点如下：**

​	1.队列中的消息会分流给组内的不同消费者，而不是重复消费，从而加快消息处理的速度

​	2.消费者组会维护一个标示，记录最后一个被处理的消息，哪怕消费者宕机重启，还会从标示之后读取消息。确保每一个消息都会被消费。

​	3.消费者获取消息后，消息处于pending状态,并存入一个pending-list。当处理完成后需要通过XACK来确认消息，标记消息为已处理，才会从pending-list移除，因此在消息执行错误之后可以从pending-list中取到。
